## §2 — Architecture ruling: A (lib/ + _core/ mirror)

**Decision:** Option A. Pure §2 evaluator and §4 roll-up live in `lib/jurisdiction/parcelHealth/` and are mirrored into `supabase/functions/parcel-health/_core/` by an extended `build:edge-core` script. Edge orchestration in `supabase/functions/parcel-health/index.ts` imports from `_core/`.

**Why A:**

- **The determination explicitly wants §7.5 boundary tests at the 2-consecutive flip, the ≤10s latency boundary, and the http→shape→latency evaluation order.** Those boundaries are exactly the kind of off-by-one and ordering bugs that silently break a gate. `run_tests` is where the rest of the codebase exercises that class of invariant. Putting parcel-health logic anywhere `run_tests` can't reach forfeits the precedent and forfeits the diagnostic surface a future regression would otherwise hit first.
- **rtc-refresh already established this exact pattern.** The `_core/` mirror with `build:edge-core` regeneration is a known, working discipline in this workspace. Slice 2 is not the moment to invent a parallel testing harness for the Edge runtime (Deno test) when the existing one covers the logic.
- **The C-6 sync cost is real but bounded.** Every PR that touches `lib/jurisdiction/parcelHealth/` must regenerate `_core/` in the same PR. This is a build-discipline tax, not an architecture risk. The CI guard that exists for rtc-refresh extends to parcel-health (see §2.3).

**Why not B:**

- A self-contained Edge Function moves the §4 roll-up (the most consequential logic — it's what flips the gate) out of `run_tests`. The Deno test harness alternative is real engineering work, parallel to a perfectly good Node/Vitest harness already in use, and any boundary regression would surface late.
- The "no `_core/` sync" upside is a one-time win that decays — every future change has to deal with the absent sync discipline by hand.

### §2.4 — Boundary test suite (§7.5) — minimum coverage

The `run_tests` suite for slice 2 must include, at minimum:

**evaluateProbe.ts:**
- HTTP 200 + valid shape + latency exactly 10000ms → healthy (≤ boundary)
- HTTP 200 + valid shape + latency 10001ms → unhealthy, reason `latency`
- HTTP 500 + valid shape + latency 5000ms → unhealthy, reason `http_status` (http precedence over shape)
- HTTP 200 + missing required field + latency 5000ms → unhealthy, reason `response_shape`
- HTTP 200 + valid shape + latency 9999ms → healthy (just under)
- HTTP 200 + invalid shape + latency 12000ms → unhealthy, reason `response_shape` (shape precedence over latency, per http→shape→latency)

**rollUpStatus.ts:**
- live + 1 fail → not_live? **NO**, still live, consecutive=1, no transition
- live + 2 consecutive fails → not_live, consecutive=2, transition `to_not_live`
- not_live + 1 success → live, consecutive=0, transition `to_live`
- not_live + 1 more fail (sustained outage) → not_live, consecutive=3, **no transition emitted** (no re-alert)
- live + fail + success → live, consecutive=0, no transition (single transient absorbed)
- per-endpoint isolation: County flipping does not touch ZIMAS row

The "no re-alert on sustained outage" invariant is the one the determination's §4 alert-on-transition rule depends on. Test it explicitly; do not rely on it emerging from the code.

## §3 — Alert sink vendor ruling: Resend

**Decision:** Resend, behind a single `AlertDestination` interface. Email is the only Tier-1 channel slice 2 wires; webhook (Tier-2) remains deferred per the live determination's sub-fork 4.

**Why Resend:**

- **Clean REST API from Deno/Edge with a single `fetch`.** No SDK bloat, no Node-specific shims, no transitive deps. The Edge runtime is the constraint and Resend is the cleanest fit.
- **Lowest integration cost on this stack.** SendGrid and SES both work but bring more API surface than slice 2 needs. Resend's surface is roughly "POST a JSON body, get back a message id" — exactly what an on-transition alert sink should be.
- **Swappable.** The whole point of the `AlertDestination` abstraction is that the vendor is interchangeable. If Resend ever degrades or pricing shifts, a new adapter behind the same interface swaps in cleanly. The ruling is for slice 2; it is not a permanent commitment to Resend.

### §3.1 — `AlertDestination` interface shape (locked)

```ts
// lib/jurisdiction/parcelHealth/alert.ts
export interface AlertEvent {
  endpoint: 'county' | 'zimas';
  transition: 'to_live' | 'to_not_live';
  detectedAt: string;      // ISO-8601 UTC
  reason?: 'http_status' | 'response_shape' | 'latency';  // present on to_not_live; absent on to_live
  context: {
    consecutiveFailures: number;
    lastSuccessAt: string | null;
    lastProbeAt: string;
  };
}

export interface AlertDestination {
  readonly kind: 'email' | 'webhook';
  send(event: AlertEvent): Promise<{ ok: true } | { ok: false; error: string }>;
}
```

The interface is the load-bearing piece. Resend lives behind it as `EmailAlertDestination` with a Resend-specific `send` implementation. A future SendGrid/SES/webhook adapter implements the same `send` signature.

### §3.2 — Env-var split (credential rail discipline)

- `PARCEL_HEALTH_ALERT_EMAIL` — recipient address. **Not a secret.** Configurable. Lives in Vercel/Supabase env config; can land in `.env.example` with a placeholder.
- `RESEND_API_KEY` — Resend API key. **Secret.** Rides the credential rail. Broker-held, injected at deploy, never in git, never in CI logs, never in Vercel client-side env. Same posture as `SUPABASE_RTC_READER_JWT` and `SUPABASE_SERVICE_ROLE_KEY`.

Reminder: if the broker-held key ever needs rotation, rotate at Resend → update the secret in the deploy injection path → no code change needed.

### §3.3 — Alert-on-transition discipline

- Fire on `to_not_live` and `to_live` only.
- Do NOT fire on `live → live`, `not_live → not_live`, or any non-transition state.
- The roll-up function returns the transition (or `null`); the orchestrator only invokes `AlertDestination.send` when transition is non-null.
- If `AlertDestination.send` returns `{ ok: false }`, log the error (probe-results-side note or separate alert-failure log) but do NOT block the probe cycle, do NOT retry the email synchronously, do NOT re-fire on the next cycle (the transition has already happened; the gate state is correct regardless of whether the email landed). Slice 3 can add an alert-delivery retry queue if operational reality demands it.
