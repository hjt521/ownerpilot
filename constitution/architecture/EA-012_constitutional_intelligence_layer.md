# EA-012 — Constitutional Intelligence Layer™ (PROPOSED · scope mandate)

> **Recorded, not designed.** The Founder directed that the Constitutional Intelligence Layer be formalized as a **first-class Enterprise Architecture document**, not only a doctrine. This file records the **required scope** of EA-012 so the mandate is canonical; the architecture itself is **Architecture-Draft work not performed here** and is not authorized to begin until the Founder says so. The `doctrines/constitutional_intelligence_layer.md` doctrine remains the ratified *principle*; EA-012 will be the ratified *architecture* that implements it.

## Purpose
Define the single architecture that **all** intelligence models inherit, so Behavioral, Trust (TM-001), Confidence (CM-001), Negotiation, Decision, and future models are consistent, interoperable, and governable — never independent scoring systems.

## Required definitions (Founder-specified scope — EA-012 must define each)
1. **What an intelligence model is** — the canonical definition and boundary (vs a plain ML model or a heuristic).
2. **How models are registered** — via the Intelligence Model Registry (see `roadmap/intelligence_model_registry_proposal.md`); canonical ID, ownership, versioning.
3. **How models consume evidence** — the shared evidence model (classes, provenance, `Indeterminate` when insufficient); AI interpretation never represented as source evidence.
4. **How models produce assessments** — the common assessment shape (subject, context, decision-use, dimensions, evidence items, contradictions, missing evidence, confidence, output, valid_until, assessor, model_version, supersedes).
5. **How models are evaluated** — evaluation-suite contract (adversarial/edge scenarios; reproducible; versioned with the model).
6. **How models are audited** — CA-001 is the common assurance path; AI output is never ratification.
7. **How models interoperate** — reference rules (e.g. TM-001 references CM-001 for evidence-confidence; models cite, they do not duplicate) — carried by the P5.5 Knowledge Graph.
8. **How models are versioned** — semantic versioning, `supersedes`/`superseded_by`, and lifecycle state per the artifact-lifecycle standard.

## Constitutional constraints (inherited)
Shared evidence/traceability/review foundation · no independent scoring systems · no hidden adverse scoring · no universal cross-context score · no material adverse action from a single model · no autonomous AI ratification · every conclusion evidence-backed and explainable.

## Placement & dependencies
EA-012 is the architecture that the P2–P5.5 foundation phases feed: Canonical Architecture Mapping (P2) inventories the objects, the Knowledge Library (P3) and metadata (P4) describe them, the Capability Registry (P5) registers capabilities, the Knowledge Graph (P5.5) connects them — and EA-012 ties these into the model architecture. **Ratify EA-012 before TM-001 (P6) design begins**, so TM-001 is the first model to inherit the finished layer.

## Status in program
**Proposed** (lifecycle: Proposed) · scope mandate recorded · **no architecture, no schema, no DB change, no ratification** until the Founder authorizes EA-012 design and later ratifies the completed document.
